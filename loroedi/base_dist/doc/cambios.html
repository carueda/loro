Resumen de cambios
Versiones 0.8preX
Loro 0.8 será una versión completamente renovada del sistema en comparación con las versiones 0.7 y anteriores, sobre todo en el aspecto de interfaz gráfica de usuario. Durante el proceso de su desarrollo se estarán ofreciendo versiones beta 0.8preX que irán agregando las características deseadas y que permitirán llevar a cabo las pruebas pertinentes.

0.8pre4-0.8pre8  - 2002-11-27
  • Sobre nuevo entorno:
    • Nuevo concepto de "demostración" asociado a un proyecto. Una demostración es un "guión" de código Loro tratado como un cuerpo integral. Se tienen las nuevas opciones "editar demo" y "ejecutar demo" (Ctrl-M).
    • Complementada opción "Exportar proyecto". Ahora se puede exportar a un directorio o a un archivo proyecto. Este archivo proyecto (extensión .lar) puede cargarse ("instalarse") fácilmente desde en otro sistema Loro.
    • Complementada opción "instalar proyecto" para incluir, además de proyectos incluidos con el sistema, otros de proveniencia externa, es decir, desde la web o desde el sistema local de archivos.
  • Núcleo del lenguaje
    • Permitida ahora una lista de acciones como entrada para interpretación interactiva (antes sólo una acción era permitida). Ej:

      •      $ cad: cadena := "hola "; cad := cad + "mundo"; escribir(cad); cad
    • Algunos avances en manejo de métodos y otras características orientadas a objetos, pero aún incompleto. Luego se notificarán oficialmente.
    • Agregada posibilidad de implementar algoritmos utilizando Java a través de BeanShell.
  • Sobre Compatibilidad
    • A nivel de fuentes:
      1. Hay nuevas palabras reservadas para manejo de excepciones: "lance", "intente", "atrape", "siempre". También se ha agregado "interfaz" que es la palabra correcta según la RAE.  Por ahora se deja también como sinónimo "interface".
      2. Ahora todo acceso a los atributos de una clase se debe hacer a través de la palabra reservad éste, tanto en constructores como en métodos. Anteriormente las declaraciones de los atributos seguían vigentes tanto en los contructores como en los métodos. Un efecto de este cambio es que ahora es posible declarar parámetros repitiendo nombres de atributos.
    • A nivel de binarios: Hay cambios internos. Para actualizar un proyecto, revisar primero posibles incompatibilidades a nivel de fuentes (según punto anterior) y luego dar el comando F9 (compilar).
  • Diversas anomalías corregidas.
0.8pre3  - 2002-10-08
  • Sobre nuevo entorno:
    • Mejorados diversos controles para definición de elementos de composición de los proyectos.
    • Habilitada la escogencia de una especificación teniendo en cuenta todos los proyectos del espacio de trabajo.
    • Mejorado despliegue del diagrama, así como algunos controles internos (antes era posible que se repitieran algunos elementos).
    • Actualización de enfoque en listas de selección.
    • Confirmación antes de salir del entorno.
    • Nueva ventana para mostrar la variables declaradas en el entorno de ejecución (Ctrl-E). Estas variables son las que se declaran en el intérprete interactivo o en ventanas de ejecución (ver a continuación). El contenido de esta ventana se actualiza ante nuevas declaraciones y borrado de declaraciones. También se ha incluido un primer criterio de actualización automática ante compilaciones que puedan comprometer las variables actualmente declaradas (en cuyo caso se borran).
    • Unificación de las ventanas de ejecución. Ahora siguen el mismo estilo del intérprete interactivo. En particular, Ctrl-F9 (ejecutar algoritmo), permite ahora acceder a las variables declaradas en el entorno para fines de suministrar argumentos y/o asignar valores de resultado, si es necesario.
    • Implementadas las opciones "Crear esquema para pruebas" (para especificación), "Probar" (para algoritmo), y "Probar proyecto" (Ctrl-T) para todo un proyecto. Esta última, por ejemplo, pone a prueba automáticamente todos los algoritmos del proyecto que cuenten con esquema de pruebas.
    • Mejorado mecanismo de compilación de un proyecto completo ante posibles interdependencias entre las unidades constituyentes.
  • obtAlgoritmo(nombre: cadena) ahora retorna nulo si el algoritmo no se encuentra (antes se generaba un error en ejecución).
  • Operador 'implementa' habilitado para revisar si una expresión es un algoritmo que implementa una cierta especificación. Ejemplo:

    •     alg: algoritmo := fiboIter;
        ...
        si alg implementa algoritmo para fibo entonces
            f := (alg como algoritmo para fibo)(20);
        ...
  • Sobre Compatibilidad
    • A nivel de fuentes: Hay dos cambios:

      • (1) Se permite ahora una cadena de documentación (que se escribe encerrada entre pares de comillas sencillas) en el contenido de una afirmación. Esta se vuelve la forma preferida en vez de cadena literal (comillas dobles). Por compatibilidad, se seguirá aceptando temporalmente la cadena literal hasta que se avise lo contrario. Ejemplo: Modificar {"duerme por 5 segundos"}  por  {''duerme por 5 segundos''}.
      (2) Ahora se acepta sólo una comilla sencilla (') al final de un identificador para fines de indicar una variable semántica. Si se utiliza algo como n'', será necesario modificar el código fuente.
    • A nivel de binarios: Hay cambios internos. Sólo bastará un F9 (compilar) para cada proyecto que deba actualizarse. Los paquetes de extensión será marcados con un aviso respectivo.
  • Anomalías corregidas:
    • 'continue' dentro de iteración 'para' no estaba actualizando variable de control. Corregido.
    • Posible persistencia de declaraciones internas y/o etiquetas en interpretación interactiva. Corregido.
    • Posible conversión inválida de valor de retorno en algoritmos. Corregido.

0.8pre2  - 2002-08-31
(Primera 0.8preX anunciada)
  • Totalmente nueva interface gráfica, mucho mejor integrada, intuitiva y funcional.
    • Manejo por proyectos.
    • Diagrama para mostrar componentes de un proyecto (estilo UML).
    • Interacción de usuario más cómoda y supervisada.
    • Actualización automática de documentación HTML.
    • Compilación de un proyecto completo o por unidad.
    • Importación/Instalación/Exportación de proyectos.
    • Las demostraciones vienen ahora como proyectos. El comando "Instalar" permite poner las demostraciones en el espacio de trabajo.
    • Comandos de teclado reasignados buscando más estandarización.
    • Reasignados colores para algunos elementos léxicos del lenguaje para mejor diferenciación.
  • ¿Y mis programas anteriores?

    • Este cambio a nivel del entorno obliga a que los programas existentes sean importados al entorno en forma de proyectos. Por cada conjunto de programas fuentes que se quiera tener como un proyecto, se debe seguir este procedimiento general:
    • Utilizar el comando "Nuevo..." para definir el proyecto. Llenae los campos informativos que se piden.
    • Para cada uno de los archivos fuentes, utilizar el comando "Importar". Indicar el archivo y Loro lo cargará y asociará al proyecto. Por el momento Loro sólo puede importar archivos que son válidos (sin errores de compilación). Si este no es el caso, de todas maneras se pueden definir las unidades (especificaciones, algoritmos o clases) utilizando el entorno, y luego complementar el código fuente de dichas unidades mediante el mecanismo copiar-pegar.
  • Ahora se revisan las expresiones que se usan como acciones: Ya que una expresión puede ser una acción, ahora se revisa que dicha expresión sea permitida. Las expresiones permitidas como acciones son:
      • Asignación
      • Invocación
      • Creación de arreglo
      • Creación de objeto
      • Condición (operador ternario ?:)
  • Nuevo elemento léxico para textos de documentación. Ahora un texto de documentación debe encerrarse en pares de comillas sencillas, por ejemplo:

    •     ''Este es el texto de documentación''
    Esto permite reflejar explícitamente la diferencia lógica que hay entre un texto de documentación y un literal cadena.  Sin embargo, por compatibilidad con código anterior, aún se siguen aceptando las comillas dobles.
  • Nuevo elemento léxico para marcas {{escriba aquí}}. Sintácticamente este elemento no es permitido en ninguna parte del programa, pero es una forma de indicar áreas concretas que deberían remplazarse por código válido.
  • Versión cadena de objetos modificada para hacer mejor control de posible recursión infinita. Un ejemplo con arreglos:
    • arr : [] Objeto;  // un arreglo de objetos
    • obj : Objeto;     // un objeto
    • arr := ["hola", "mundo'];
    • obj := arr;       // válido porque los arreglos son objetos
    • arr[1] := obj;    // válido, pero note el ciclo recursivo que se forma
    Si se imprime obj (o arr, pues son el mismo objeto), se obtiene:
      ["hola",[!]]
    el símbolo [!] indica que se trata de un arreglo que ya se se está procesando (o ya se procesó). Para objetos que son miembros de otros objetos, se utiliza el símbolo {!}.
    Los comandos .verobj para establecer el "máximo nivel para ver objetos", y .verarr para la "máxima longitud para ver arreglos", siguen operando de la misma manera.
  • Sobre Compatibilidad
    • A nivel de fuentes: Hay cambios pero hay compatibilidad. Sin embargo, es altamente recomendable remplazar las comillas dobles para textos de documentación por pares de comillas sencillas. Ejemplo: Modificar "bla bla bla" por ''bla bla bla''.

      • Nota: las cadenas literales de caracteres (cuyo tipo es cadena) continúan escribiéndose con comillas dobles.
    • A nivel de binarios: Hay cambios internos. De hecho, el cambio general del entorno obliga a que los programas existentes sean importados al entorno en forma de proyectos. Ver más arriba.
  • Sobre requerimientos:
    • El entorno Loro requiere ahora un ambiente de ejecución Java (JRE) version 1.4 o posterior.
  • Anomalías corregidas:
    • Una definición de clase A recursiva (con un miembro de clase A), con un constructor asignando al miembro: producía error de compilación asociado a la NO resolución del tipo recursivo. Corregido.
    • Conversión de valor operaba mal en el caso 'nulo' (el cual debe dejarse tal cual). Corregido.
    • obj.xxx() generaba error de invocación sobre objeto nulo. Corregido. (Esta anomalía surgió recientemente por la adición de ciertos mecanismos para el manejo de objetos y métodos, los cuales aún no se encuentran habilitados.)
    • Posible recursión infinita al visualizar un objeto o un arreglo. Corregido. (Ver arriba.)

0.7s1  - 2002-04-25
  • SOBRE COMPATIBILIDAD
    • Nuevo mecanismo para verificación de posibles problemas de compatibilidad con objetos compilados existentes.  Esta verificación se realiza de manera automática al inicio de la primera ejecución de esta versión del sistema. Si la verificación no detecta ningún problema, el sistema se inicia de la manera acostumbrada, por lo que no se percibirá ningún cambio. 0.5 - Marzo 1 de 2001
  • Ahora se puede escoger para ejecución cualquiera de los algoritmos "ejecutables directamente" disponibles en la última compilación exitosa. Un algoritmo "ejecutable directamente" es aquel que o bien no recibe entradas, o bien las entradas son de tipos básicos (entero, caracter, real, cadena, etc.).
  • Ahora se ofrece un mejor control sobre los algoritmos en ejecución:
    • Se destina una consola por cada ejecución
    • Se puede detener una ejecución en cualquier momento (antes había que salirse del sistema para lograr esto).
  •  Nueva versión de consola más estándar (es decir, más similar a una ventana de "shell" o de DOS. Esta consola permite recordar últimas entradas (a la manera del bash o del DOSKEY). También puede cambiársele el tamaño y estilo de la letra.
  • La zona de mensajes dejó de ser en ventana aparte pasando a ser parte (inferior) de la ventana principal.
  • Se quitó menú "Ver" con las opciones "Consola" y "Mensajes" (ver puntos anteriores).
  • BUG corregido: no se detectaba en tiempo de ejecución si un identificador tenía valor asignado o no; por ejemplo, se imprimía ''nulo'' para tipos primitivos cuando sucedia esto.
  • Se ha hecho una migración completa al manejador "Swing" de interfaz gráfica ofrecido por el entorno Java. Esto se ha complementado con mejores características de navegación y visualización.
  • Nuevo proceso incorporado dormir(milisegundos) para poner a dormir la ejecución durante un determinado tiempo.
  • Nuevo proceso incorporado terminarEjecucion(cód: entero) que permite terminar la ejecución en cualquier punto (al estilo de otros lenguajes: exit(cód) en C y System.exit(cód) en Java). NOTA: Se llama terminarEjecucion, SIN tilde (debido a un bug de Java/zip).
  • Los archivos compilados pueden quedar de tamaños moderadamente grandes cuando hay referencias mutuas entre la unidades de compilación, particularmente cuando se tienen algoritmos que se llaman recursivamente entre sí. Esto puede manifestarse por algunas demoras en el momento de la compilación.
  • Nuevo manejo de "elementos pendientes" en la fase de compilación, lo que permite el uso de identificadores de unidades (algoritmos, especificaciones, clases) antes de que éstos sean definidos más adelante en el mismo archivo fuente.
  • BUG corregido: Se admitía en compilación invocar una especificación. Esto generaba desde luego un error en ejecución.
  • El posible error de desbordamiento de pila ("stack overflow") se reporta ahora mostrando la propia pila de ejecución de Loro (y no la de Java).

0.4 - Febrero 12 de 2001
  • En esta versión se requiere un entorno Java versión JDK 1.3 (hasta la versión anterior se requería JDK 1.1.6). Esto debido principalmente a que se han empezado a utilizar clases Swing para la elaboración de la interfaz gráfica de usuario.
  • Nueva característica de resaltado de sintaxis. Ahora será mucho más fácil visualizar la estructura de un programa y revisar y ubicar "a ojo" posibles errores de compilación.
  • Nuevo menú con opciones para edición: Cortar, Copiar, Pegar, Seleccionar todo, Buscar, Buscar siguiente, Ir a una línea. Algunas de estas opciones tienen forma de operarse por teclado; el comando de ayuda,  F1, da una guía sobre esto.
  • Corregido error al ejecutar acciones para cuando a la variable de la iteración se le hacía una asignación directa dentro del cuerpo.
  • Nueva acción ciclo para indicar iteración. Su esquema básico es:

    •        ciclo
               acciones
            fin ciclo;
  • Nuevas acciones termine y continúe que permiten manejar el flujo de control dentro de las acciones de una iteración. Una acción termine provoca la terminación de la iteración más inmediata en la que se encuentre ubicada. Una acción continúe provoca que el flujo de control vaya directamente al inicio del cuerpo de la iteración (previa verificación de la condición de permanencia, en caso distinto a un ciclo). Estas nuevas acciones permiten opcionalmente una condición para provocar su efecto; por ejemplo:

    •         termine si i > 21;
    terminaría la iteración más interna en caso que i sea mayor que 21.
  • Todos los mecanismos de iteración (ciclo, para, mientras, repita) pueden ahora etiquetarse como medio de identificación para efectos de las acciones termine y continúe. Una etiqueta se indica con un identificador encerrado entre los símbolos  '<'  y  '>'.  Por ejemplo:

    •        ciclo <externo>
              acciones
              para j desde m hasta h haga
               otras_acciones
                  termine <externo> si alguna_condición;
               mas_acciones
              fin para;
           fin ciclo;

0.38 - Agosto 16 2000
  • La variable de ambiente para indicar en donde está instalado el sistema es LORO_DIR. (Antes era LORODIR) Esta variable ahora debe estar definida pues permite ubicar el archivo de libreria loroI.lar (LORO_DIR/lib/loroI.lar).
  • Se maneja ahora la variable de ambiente LORO_RUTA_UNIDADES en donde se pueden indicar los directorios o archivos .zip a revisar para resolver nombres.
  • Ahora las rutinas de entrada y salida básicas como 'escribir' (antes 'escriba'), 'leerEntero', etc., son algoritmos normales (no mecanismos internos). Estos procesos están implementados en Java y se alojan en la nueva librería 'loroI.lar', ubicado en el subdirectorio lib de la instalación.
  • La nueva herramienta loroij permite generar el esquema en Java para la implementación de un algoritmo Loro.
  • Ya no es obligatoria la palabra "clase" para un tipo clase. Así, en vez de:

    •        p: clase Persona;
    se puede escribir:
           p: Persona;
    Sin embargo, para conversión de tipo sí es necesario utilizar el cualificador "clase" para resolver la situación.
  • A cambio de leerAlgoritmo(ID) (que ha desaparecido), se ha definido el siguiente proceso (en el paquete loroI::sistema):

    •        especificacion obtAlgoritmo(nomAlgoritmo: cadena) -> alg: algoritmo
               descripcion "Obtiene un algoritmo dado su nombre."
               entrada nomAlgoritmo: "Nombre completo del algoritmo a leer."
               salida alg: "Algoritmo correspondiente al nombre."
          fin especificacion
    Esto se complementa con el mecanismo de promocion de tipos para fines de asignacion, ej:
            alg: algoritmo;
            afibo: algoritmo para fibo;
            nom: cadena;
            ...
            nom := leerCadena();
            alg := obtAlgoritmo(nom);
            afibo := algoritmo para fibo(alg);
            f := afibo(5);
     Nótese que se admite declarar un variable como de tipo algoritmo sin decir para cuál especificación.
  • Nuevo manejo de paquetes (al estilo Java); y se ha iniciado la definicion de algunos paquetes Loro:

    •         loroI::sistema      Servicios del sistema
            loroI::mat          Funciones matematicas
            loroI::entsal       Servicios de entrada/salida
  • Las unidades (especificaciones, algoritmos, clases) del paquete loroI::sistema son visibles automáticamente (no es necesario hacer los 'utiliza' correspondientes).
  • Para facilitar la escritura de algunas afirmaciones, se permite ahora que una condición sea una cadena literal (la cual se asume inmediatamente verdadera). Ej:

    •        { "p = n-esimo numero primo" }
  • Ahora se pueden imprimir objetos:

    •        p: Persona := ...
           escribirln("p = " +p);
    Se hace hasta un cierto nivel (2) para evitar, entre otras cosas, recursion infinita.
  • Ante un error en ejecución, se reporta ahora la pila de ejecución  de Loro (que es mucho más clara desde luego que la pila de Java).
  • Solucionado bug (StackOverflow) que sucedía algunas veces al guardar unidades compiladas en disco.
  • Solucionado bug que sucedía algunas veces al compilar algunos fuentes un poco extensos (errores sintácticos que no habían).
